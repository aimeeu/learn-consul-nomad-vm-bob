# HashiStack Server and Client Configuration with Ansible

This directory contains Ansible playbooks for configuring HashiCorp Consul and Nomad servers and clients, converted from the original bash user-data scripts.

## 📁 Directory Structure

```
ansible/
├── ansible.cfg                          # Ansible configuration
├── inventory.ini.example                # Example inventory file
├── setup-hashistack.yml                 # Install HashiStack packages
├── configure-servers.yml                # Configure Consul/Nomad servers
├── configure-clients.yml                # Configure Consul/Nomad clients
├── templates/                           # Jinja2 configuration templates
│   ├── consul-server.hcl.j2
│   ├── consul-client.hcl.j2
│   ├── nomad-server.hcl.j2
│   ├── nomad-client.hcl.j2
│   ├── consul-server-tokens-bootstrap.hcl.j2
│   └── systemd-resolved-consul.conf.j2
├── policies/                            # ACL policies and configurations
│   ├── policy-nomad-tasks.hcl
│   └── jwt-auth-method-config.json
└── README.md                            # This file
```

## 🚀 Quick Start

### 1. Prerequisites

- Ansible >= 2.9 installed
- SSH access to target hosts
- HashiStack packages already installed (run `setup-hashistack.yml` first)
- Required environment variables or Ansible Vault configured

### 2. Setup Inventory

```bash
cp inventory.ini.example inventory.ini
# Edit with your host IPs and SSH configuration
```

### 3. Configure Environment Variables

Export required secrets (or use Ansible Vault):

```bash
# Encryption keys (generate with: consul keygen / nomad operator keygen)
export CONSUL_ENCRYPTION_KEY="your-consul-encryption-key"
export NOMAD_ENCRYPTION_KEY="your-nomad-encryption-key"

# Management tokens (generate with: uuidgen or consul acl bootstrap)
export CONSUL_MANAGEMENT_TOKEN="your-consul-management-token"
export NOMAD_MANAGEMENT_TOKEN="your-nomad-management-token"

# Client tokens (generated by server playbook, or pre-created)
export CONSUL_AGENT_TOKEN="your-consul-agent-token"
export CONSUL_DEFAULT_TOKEN="your-consul-default-token"
export NOMAD_AGENT_TOKEN="your-nomad-agent-token"

# TLS Certificates (base64 encoded, gzipped)
export CA_CERTIFICATE="base64-encoded-gzipped-ca-cert"
export AGENT_CERTIFICATE="base64-encoded-gzipped-agent-cert"
export AGENT_KEY="base64-encoded-gzipped-agent-key"
```

### 4. Run Playbooks

**Step 1: Install HashiStack packages** (if not already done)
```bash
ansible-playbook setup-hashistack.yml
```

**Step 2: Configure servers**
```bash
ansible-playbook configure-servers.yml
```

**Step 3: Configure clients**
```bash
ansible-playbook configure-clients.yml
```

## 📋 Playbook Details

### setup-hashistack.yml

Installs HashiStack components on all nodes:
- Consul v1.22.5
- Nomad v2.0.1
- Vault v1.21.3
- Consul Template v0.41.4
- Docker CE
- OpenJDK 8
- System utilities

**Usage:**
```bash
ansible-playbook setup-hashistack.yml
```

### configure-servers.yml

Configures Consul and Nomad servers with:
- TLS certificate deployment
- Consul server configuration with ACLs
- Nomad server configuration with ACLs
- Consul-Nomad workload identity integration
- DNS configuration via systemd-resolved
- ACL token generation
- Nomad ACL bootstrap (first server only)
- Consul auth-method and binding rules setup

**Key Features:**
- ✅ Automatic IP address detection (AWS/GCE/Azure)
- ✅ TLS encryption for Consul and Nomad
- ✅ ACL system with token generation
- ✅ Workload identity integration
- ✅ DNS forwarding to Consul
- ✅ Idempotent operations

**Usage:**
```bash
ansible-playbook configure-servers.yml
```

**Variables:**
- `datacenter`: Consul datacenter name (default: "dc1")
- `domain`: Consul domain (default: "global")
- `cloud_env`: Cloud provider (aws/gce/azure)
- `server_count`: Number of servers (default: 3)

### configure-clients.yml

Configures Consul and Nomad clients with:
- TLS certificate deployment
- CNI plugins installation (for service mesh)
- Consul client configuration with ACLs
- Nomad client configuration with ACLs
- DNS configuration via systemd-resolved
- Docker privileged mode enabled
- Raw exec driver enabled

**Key Features:**
- ✅ CNI plugins for Consul Connect
- ✅ Consul CNI plugin installation
- ✅ Auto-encrypt TLS for clients
- ✅ Workload identity support
- ✅ Node metadata support

**Usage:**
```bash
ansible-playbook configure-clients.yml
```

**Variables:**
- `datacenter`: Consul datacenter name (default: "dc1")
- `domain`: Consul domain (default: "global")
- `cloud_env`: Cloud provider (aws/gce/azure)
- `nomad_agent_meta`: Node metadata (optional)

## 🔐 Security Configuration

### TLS Certificates

Certificates must be provided as base64-encoded, gzipped content:

```bash
# Generate and encode certificates
cat ca.pem | gzip | base64 > ca.b64
cat agent.pem | gzip | base64 > agent.b64
cat agent-key.pem | gzip | base64 > agent-key.b64

# Export as environment variables
export CA_CERTIFICATE=$(cat ca.b64)
export AGENT_CERTIFICATE=$(cat agent.b64)
export AGENT_KEY=$(cat agent-key.b64)
```

### ACL Policies

#### policy-nomad-tasks.hcl

Grants Nomad tasks permissions to:
- Read keys and nodes
- Write services to Consul

Used by the `nomad-default-tasks` role.

#### JWT Auth Method Configuration

Configures Consul to accept Nomad workload identities for authentication.

**Key components:**
- JWKS URL: `https://127.0.0.1:4646/.well-known/jwks.json`
- Supported algorithms: RS256
- Bound audiences: consul.io
- Claim mappings for Nomad metadata

## 🔄 Workload Identity Integration

The server playbook automatically configures Consul-Nomad workload identity integration:

1. **Auth Method**: Creates `nomad-workloads` JWT auth method
2. **Binding Rules**: 
   - Services: Binds to service identity
   - Tasks: Binds to `nomad-{namespace}-tasks` role
3. **Policy**: Creates `policy-nomad-tasks` for task permissions
4. **Role**: Creates `nomad-default-tasks` role with the policy

This allows Nomad jobs to automatically obtain Consul ACL tokens without manual token management.

## 🌐 DNS Configuration

Both playbooks configure systemd-resolved to forward DNS queries for the Consul domain to Consul's DNS server:

- Consul DNS: `127.0.0.1:8600`
- Domain: `~{domain}` (default: `~global`)
- DNSSEC: Disabled

**Test DNS:**
```bash
dig @127.0.0.1 -p 8600 consul.service.dc1.global
```

## 🔍 Troubleshooting

### Certificate Issues

If certificate deployment fails:
```bash
# Verify certificates are properly encoded
echo $CA_CERTIFICATE | base64 -d | zcat | openssl x509 -text -noout
```

### ACL Token Issues

If ACL operations fail:
```bash
# Check Consul is running
systemctl status consul

# Verify management token
consul members -token=$CONSUL_MANAGEMENT_TOKEN
```

### Network Issues

If nodes can't join:
```bash
# Check retry_join configuration
consul members

# Verify security groups allow required ports
# Consul: 8300, 8301, 8302, 8500, 8600
# Nomad: 4646, 4647, 4648
```

### DNS Issues

```bash
# Check systemd-resolved status
systemctl status systemd-resolved

# Verify DNS configuration
resolvectl status

# Test Consul DNS
dig consul.service.dc1.global
```

## 📝 Using Ansible Vault

For production, use Ansible Vault to encrypt sensitive variables:

```bash
# Create vault file
ansible-vault create group_vars/all/vault.yml

# Add encrypted variables
consul_encryption_key: "your-key"
nomad_encryption_key: "your-key"
consul_management_token: "your-token"
nomad_management_token: "your-token"

# Run with vault
ansible-playbook configure-servers.yml --ask-vault-pass
```

## 🆚 Comparison with Bash Scripts

### Advantages of Ansible Approach

✅ **Idempotent**: Safe to run multiple times
✅ **Parallel Execution**: Configures multiple hosts simultaneously
✅ **Better Error Handling**: Detailed error reporting and recovery
✅ **Modular**: Easy to customize and extend
✅ **Templating**: Jinja2 templates for dynamic configuration
✅ **Secrets Management**: Integration with Ansible Vault
✅ **Testing**: Can use `--check` mode for dry runs
✅ **Logging**: Comprehensive execution logs
✅ **Reusable**: Templates and tasks can be shared across projects

### Original Bash Scripts

The original `user-data-server.sh` and `user-data-client.sh` scripts are designed for cloud-init and run once during instance initialization. These Ansible playbooks provide the same functionality but with better operational characteristics for ongoing management.
